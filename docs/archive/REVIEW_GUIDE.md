## 1. Введение и цели гайда

Этот документ — **живой инструмент** для систематического ревью и улучшения пайплайна обработки оптических данных сердца (оптическое картирование, APD, conduction velocity, alternans).

**Цели:**
- Сохранить и развить сильные стороны текущей реализации (файловая дисциплина, поддержка двух красителей, provenance в filled_hybrid).
- Систематически устранить критические проблемы (в первую очередь coherence filter).
- Перевести пайплайн на устойчивую, тестируемую и легко расширяемую архитектуру.
- Обеспечить continuity ревью между сессиями (чтобы не начинать каждый раз с нуля).

**Как использовать:**
1. Перед каждой сессией ревью прочитать разделы 4 и 10.
2. Работать по чек-листам раздела 5.
3. Все находки и решения **только append** в раздел 9 (Review Log).
4. После сессии обновить дату в шапке и, при необходимости, чек-листы.

## 2. Текущая архитектура пайплайна (Stages 1–6)

(На основе оригинального анализа)

- **Stage 1**: Preprocessing (motion correction, baseline, loading двух красителей A/B).
- **Stage 2**: `apd_agent.py` — первичный расчёт APD.
- **Stage 3**: `apd_worker.py` — детальный расчёт APD (highest priority для ревью).
- **Stage 4**: `conduction_analysis.py` + `source_cv_agent.py` — conduction velocity (Structure Tensor + radial profile + local_fit + filled_hybrid).
- **Stage 5**: `alternans_worker.py` — пространственный + временной alternans + concordance.
- **Stage 6**: Postprocessing, статистика, визуализация, экспорт.

**Сильные стороны (подтверждено):**
- Отличная 5-классная файловая система.
- Хорошая поддержка dye A (VSD) и dye B (Ca²⁺).
- Зрелая реализация conduction с provenance.
- Экспериментальные скрипты (`_exp_st_no_coh.py`, `_viz_016A_post_patch.py`) отлично документируют историю багов.

## 3. Файловая контрактная система (5 классов)

**Обязательно соблюдать во всех новых и рефакторенных модулях:**

| Класс          | Назначение                              | Примеры использования                  | Правила |
|----------------|-----------------------------------------|----------------------------------------|---------|
| **MUST**       | Финальные результаты, обязательные для downstream | APD maps, CV vectors, alternans metrics | Чистые имена, без суффиксов класса |
| **DEBUG**      | Диагностика, карты качества, промежуточные визуализации | coherence_map, valid_mask, fraction_kept | Суффикс `_debug`, sidecar .json |
| **INTERMEDIATE** | Временные результаты между стадиями   | filtered_traces, activation_maps       | Суффикс `_intermediate` |
| **META**       | Метаданные, provenance, параметры     | sidecar .json, params.yaml             | Всегда рядом с MUST |
| **EXTERNAL**   | Внешние данные / импорты                | raw .tif, исходные видео               | Не трогаем структуру |

**Инструменты (уже реализованы в skeleton):**
- `utils/file_contract.py` — `FileClass` enum + `save_array()` с автоматическим sidecar provenance.

## 4. Пошаговый процесс ревью (как работать с этим гайдом)

1. **Подготовка сессии**
   - Прочитать текущий Review Log (раздел 9).
   - Выбрать приоритетный модуль из чек-листа (обычно apd_worker или conduction).
   - Указать sample (например 016A) и какие файлы уже существуют.

2. **Во время ревью**
   - Идти строго по чек-листу модуля.
   - Фиксировать **только факты** + ссылки на файлы/коммиты.
   - При нахождении проблемы — сразу предлагать решение + приоритет.

3. **После сессии**
   - Append-only запись в раздел 9 (дата, что сделано, что осталось, принятые решения).
   - Обновить дату "Последнее обновление".
   - Если изменилась архитектура — обновить разделы 2, 7 или 8.

4. **Между сессиями**
   - Никогда не начинать с чистого листа — всегда читать последний лог.
   - Если меняется исполнитель — новый человек сначала читает весь гайд + последние 3 записи лога.

## 6. Известные критические проблемы (с историей)

1. **Structure Tensor coherence filter** (самая большая)
   - Оригинал: `coh > 0.1` + inner → 88–94% потеря валидных пикселей (016A и др.).
   - Патч: `integration_sigma=4.0`, `cap=5.0`.
   - Текущее решение: `filled_hybrid` (ST + gradient fallback).
   - Статус: Частично решено в новом `cv_estimators.py`. Требует валидации на реальных данных.

2. **Дублирование логики APD**
   - `apd_agent.py` (Stage 2) и `apd_worker.py` (Stage 3) частично дублируют функциональность.
   - Решение: единая `compute_apd()` в `utils/signal.py`.

3. **Мелкие inconsistencies**
   - Терминология dye=B.
   - `PIXEL_SIZE_MM` не везде централизован.
   - Двойной подсчёт ASLS baseline.

4. **Архитектурные пробелы (решены в новом skeleton)**
   - Отсутствовал центральный runner/DAG.
   - Не было автоматических тестов + synthetic data generator.
   - Параметры были разбросаны по коду.
